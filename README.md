SIP - сервис подписок и регистраций
=====

# Функциональные возможности

Данное приложение реализует SIP-сервер на базе Erlang/OTP и библиотеки NkSIP. Приложение использует базу данных Mnesia для хранения абонентов, подписок и регистраций. 

### Аутентификация

Проверка пароля пользователя в базе данных в таблице abonent.

### Регистрация

Обработка REGISTER, сохранение контакта, времени истечения регистрации, генерация событий при регистрации, разрегистрации и запросе активных регистраций пользователя. Поддерживается несколько контактов одного абонента.

### Подписка

Обработка SUBSCRIBE, хранение подписок, отправка NOTIFY при изменении статуса, возможность отписаться. При разрегистрации пользователя его подписке сохраняются в базе, но уведомления по ним не отправляются, пока пользователь не зарегистрируется снова.

### Хранение данных

Mnesia - постоянное хранилище на диске с репликацией между узлами кластера.  
ETS-кэш - оперативная память для быстрого доступа к данным при чтении и поиске. Кэш синхронизирован с Mnesia при старте и при внесении изменений.

### Очистка истекших подписок и регистраций

Каждые 15 секунд (время по умолчанию) из базы данных удаляются подписки и регистрации с истекшим временем действия.

### Кластеризация и отказоустойчивость

1. Поддерживается работа двух узлов Erlang, объединенных в кластер.
2. Mnesia реплицирует данные на оба узла.
3. Выбор мастер ноды производится на основе переменной окружения PRIORITY, чем выше число, тем выше приоритет узла. при падении мастера другой узел становится мастером (выбор мастера производится по приоритету).
4. Мастер поднимает виртуальный IP адрес (VIP), запускает SIP-обработчик и REST API (Cowboy).
5. ETS на новом мастере восстанавливается из Mnesia.

### REST API

Доступно по адресу http://localhost:8080
1. GET /api/users - получение всех абонентов
2. POST /api/users - добавления нового пользователя по AOR
3. DELETE /api/users/:aor - удаление пользователя по AOR
4. POST /api/monitor/log - запись статистики в файл (кол-во регистраций и подписок)
5. POST /api/monitor/log/full -  запись статистики в файл (регистрации и подписки)

# Установка и запуск проекта

### Клонирование репозитория

```
git clone https://study-git.eltex-co.ru/erlang-projects/2026-july-group/sip-subscription-service.git
```

Git автоматически создаст папку с именем проекта и скачает все файлы с репозитория. Для начала работы необходимо перейти в папку проекта и открыть в ней терминал.

### Требования к окружению

Проект предназначен для запуска в Docker-контейнере с использование docker-compose. Для работы необходим Docker и docker-compose, все необходимые зависимости уже включены в образ.

### Сборка образа

```
docker build -t sip-serv .
```

### Запуск контейнера

Рекомендуется использовать сетевой режим host чтобы контейнер мог принимать SIP-запросы напрямую на IP-адрес хоста
```
docker run -it --rm --name sip-node1 --network host -e NODE_NAME="node1@127.0.0.101" -e PRIORITY="150" -e ERLANG_COOKIE="mysipsecret" -v "$(pwd)/logs:/app/logs" -v "$(pwd)/config:/app/config" sip-serv
```
Для работы в изолированной сети можно использовать проброс портов
```
docker run -it --rm --name sip-node1 -p 5060:5060/udp -e NODE_NAME="node1@127.0.0.101" -e PRIORITY="150" -e ERLANG_COOKIE="mysipsecret" -v "$(pwd)/logs:/app/logs" sip-serv
```
Для запуска второго узла неободимо открыть второй терминал и ввести в него команду
```
docker run -it --rm --name sip-node2 --network host -e NODE_NAME="node2@127.0.0.102" -e PRIORITY="100" -e ERLANG_COOKIE="mysipsecret" -v "$(pwd)/logs:/app/logs" -v "$(pwd)/config:/app/config" sip-serv
```

В обоих случаях после запуска вы попадете в интерактивную оболочку Erlang с запущенным приложением sip_serv, все логи будут выводиться в консоль. Также локально будет создана директория logs, в которую при выполнении запросов статистики будет помещен файл active.log

### Остановка и удаление контейнера

Для остановки работающего приложения используйте сочетание Ctrl+G, затем нажмите q для выхода.
Для остановки работы контейнера можно использовать сочетание Ctrl+C или команду
```
docker stop sip-serv
```
При необходимости можно удалить образ.
```
docker rmi sip-serv
```

# Архитектура проекта

Проект состоит из следующих модулей:

1. sip_serv_app - корневое приложение: создает схему Mnesia, таблицы, загружает абонентов из sys.config, запускает супервизор
2. sip_serv_sup - супервизор, запускает процесс периодической очистки таблиц - sip_serv_cleanup и sip_serv_master - процесс выбора мастера
3. sip_serv_cleanup - модуль, реализующий поведение gen_server, отвечающий за очистку истекших регистраций и подписок, вызывает функцию очистки бд каждые 15 секунд
4. sip_serv_db - модуль управлением базой данных Mnesia, отвечает за создание, удаление и поиск регистраций, подписок и абонентов
5. sip_serv_cache - ETS-кэш для быстрого доступа к данным, дублирует таблицы Mnesia
6. sip_serv_auth - модуль, отвечающий за работу с абонентами
7. sip_serv_handler - модуль-обработчик NkSIP, принимает SIP запросы, обрабатывает REGISTATION и SUBSCRIBE
8. sip_serv_master - модуль реализующий процесс проверки активных узлов и выбор мастер узла
9. sip_serv_monitor - модуль сбора статистики и ее вывода в файл active.log
10. sip_serv_api - модуль API для вызова функций API
11. sip_log_handler, sip_user_handler, sip_stats_handler – обработчики HTTP-запросов Cowboy
12. sip_serv_conf - модуль для чтения файла конфигурации, который содержит список абонентов
13. sip_serv_records - заголовочный файл, содержащий записи для формирования таблиц

Таблицы Mnesia:
1. abonent - хранит данные об абонентах (тип - set, ключ - aor)
2. registration - хранит данные об активных регистрациях (тип - bag, ключ - aor)
3. erlsubscription - хранит данные об активных подписках (тип - set, ключ - id)
4. publication - хранит данные о состояниях пользователя publish (тип - bag, ключ - aor)

# Конфигурация

Файл sys.config содержит:

- Путь к данным Mnesia - директория /tmp/mnesia (создается при сборке образа)
- Список узлов кластера
- VIP-адрес
- Список абонентов - каждая запись содержит AOR, пароль и домен

# Документация

Исходный код имеет документацию, написанную с помощью edoc. Для ее генерации необходимо собрать образ и выполнить команду
```
docker run -it --rm --name sip-serv --network host -v "$(pwd)/doc:/app/doc" sip-serv sh -c "rebar3 edoc"
```
Документация появится в каталоге doc/, которая может быть открыта только от root. Для выдачи прав на открытие файлов в папке необходимо выполнить команды
```
sudo chown -R $USER:$USER doc
chmod -R u+rwX doc
```
Для просмотра необходимо открыть файл index.html в браузере.

# Тестирование

## Twinkle

1. Установить Twinkle  
    sudo apt-get install twinkle
2. Перенести файлы из /test/twinkle в папку ~/.twinkle  
    cp -r /test/twinkle/* ~/.twinkle
3. Запустить Twinkle, выбрать необходимых клиентов, произвести регистрацию. В логах контейнера появится сообщение об успешной регистрации. Произвести ручную разрегистрацию или подождать ее истечения, появятся логи о разрегистрации. Также можно запросить список активных регистраций, будут выведены все регистрации клиента и оставшееся время их длительности.
4. Для проверки подписки необходимо зарегистрировать несколько пользователей и добавить их в друзья.

## SIPP

Скрипты для тестирования находятся в папке /test/sipp. В ней находятся сами SIPP-скрипты и скрипт на языке bash для их запуска.

### Установка SIPP

sudo apt-get install sipp

### Необходимо сделать файлы исполняемыми

Находясь в папке /test/sipp нужно открыть терминал и выполнить команду
```
chmod +x run_test.sh
```

### Выполнение скриптов

Необходимо запустить скрипт
```
./run_test.sh
```
Будут зарегистрированны два пользователя user1 и user2, далее user1 подпишется на user2, будет отправлен NOTIFY и скрипт завершится. В логах выполнения скрипта будут выведены вспомогательные сообщения о регистрации пользователей, подписки и успешном завершении выполнения тестов. В логах сервера можно увидеть работу SIP-сервера.

## Юнит-тесты (EUnit)

- Модуль sip_serv_db_tests - тестирование операций с таблицами Mnesia
- Модуль sip_serv_cache_tests - проверка работы ETS-кэша
- Модуль sip_serv_auth_tests - тестирование функций аутентификации и управления пользователями
- Модуль sip_serv_api_tests – тестирование API-функций (добавление/удаление пользователей, вызовы мониторинга)

Для запуска тестов необходимо собрать образ и выполнить команду
```
docker run --rm sip-serv rebar3 eunit
```
